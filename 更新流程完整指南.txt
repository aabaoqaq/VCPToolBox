# VCPToolBox AI 更新执行指南

> 本文件是 **AI 执行仓库更新时的唯一权威指南**。  
> 旧的人手更新说明、零散规则文件、经验性习惯，如果与本文件冲突，**以本文件为准**。

---

## 1. 目标

本指南用于约束 AI 在 `VCPToolBox` 仓库中执行“按脚本流程更新官方上游”的完整操作。

AI 的目标不是随意 `pull` 或盲目 `merge`，而是：

1. 先读取规则，再读取脚本；
2. 严格按 `自动更新.bat` 的阶段顺序等价执行；
3. 保护本地私有文件、记忆文件、配置文件；
4. 遇到指定范围的冲突时，暂停并向主人展示冲突内容；
5. 更新完成后输出 **人类可读的更新日志**；
6. 只把允许进入备份仓库的内容推送到 `origin/custom`。

---

## 2. 更新前必须先读什么

每次执行更新时，AI 必须按以下顺序检查：

1. 本文件：`更新流程完整指南.txt`
2. 更新脚本：`自动更新.bat`
3. 如果存在，则读取：`自动更新.bot`
4. 当前仓库状态：`git status --short --branch`
5. 当前未解决冲突：`git diff --name-status --diff-filter=U`

禁止跳过以上检查后直接操作。

如果 `自动更新.bat` 中使用了当前终端无法交互处理的 `choice`、`pause` 等逻辑，AI **不得因此中止任务**；应改为按照脚本的相同阶段与相同语义，手工执行等价的 git / 依赖 / 推送步骤。

---

## 3. 固定仓库信息

- 工作分支：`custom`
- 官方上游：`upstream`
- 个人备份仓库：`origin`
- 官方同步目标：`upstream/main`
- 个人备份推送目标：`origin/custom`

如果当前不在 `custom`，先切换到 `custom`，再继续更新。

---

## 4. 本地保护规则

以下规则用于防止 AI 再次误动主人的本地内容。

### 4.1 永远视为本地私有，不自动提交、不自动推送

- 根目录真实配置：`config.env`
- 所有插件真实配置：`Plugin/*/config.env`
- 本地排除清单：`.update-exclude.txt`
- 本地自定义更新脚本：`自动更新.bat`

如果这些文件出现在工作区改动里，默认保留在本地，不进入提交，不进入推送。  
除非主人明确说“可以提交 / 可以推送”，否则不能改变这条规则。

### 4.2 敏感内容目录

- `Agent/`
- `TVStxt/`

这两个目录不是普通代码目录。AI 必须按下面的细分规则处理。

### 4.3 `Agent/` 处理规则

- **纯新增文件**：可直接接受上游新增；
- **已有文件被修改**：必须暂停，展示冲突片段，再等主人决定；
- **已有文件被删除**：必须暂停，说明是本地侧删除还是上游侧删除，再等主人决定。

特别说明：

- `Agent/Nova.txt` 视为高敏感文件；
- 只要它发生非纯新增变动，就必须汇报，不允许静默覆盖。

### 4.4 `TVStxt/` 处理规则

- **纯新增文件**：可直接接受上游新增；
- **所有 `.example` 文件**：直接使用上游版本；
- **非 `.example` 的已有文件被修改或删除**：必须暂停，展示冲突片段，再等主人决定。

### 4.5 `dailynote/` 处理规则

`dailynote/` 不是普通可随意清理的缓存目录。

- 对于 **上游已跟踪的更新内容**，可以正常合并；
- 对于 **本地私有、未跟踪、未明确授权处理的记忆/日记内容**，禁止擅自删除、擅自提交、擅自推送。

如果 AI 无法区分某个 `dailynote/` 文件到底属于上游跟踪内容还是主人的本地私有内容，先汇报，再操作。

---

## 5. 更新执行总原则

1. **先保护，再合并**：先确认哪些文件不能进提交，再做更新。
2. **先展示，再决策**：对需要主人拍板的冲突，必须先给出路径、冲突类型、双方片段。
3. **纯新增优先放行**：主人已明确允许的新增项，不要反复询问。
4. **`.example` 一律跟上游**：不管位于哪里，只要是示例文件，默认接受上游版本。
5. **插件除 `.env` 外默认跟上游**：插件代码、说明、manifest、锁文件等都优先接受上游。
6. **禁止猜测**：不确定就查 git 状态、索引、冲突三方版本，不要凭印象回答。
7. **更新后必须产出说明**：不能只做完 git 操作就结束，必须整理可读更新日志。

---

## 6. 标准执行流程（严格对齐 `自动更新.bat`）

### 阶段 0：检查本地修改与保护项

AI 必须：

1. 检查当前分支是否为 `custom`；
2. 枚举所有未提交修改、暂存修改、未跟踪文件；
3. 明确列出哪些文件会被排除在提交之外；
4. 在真正开始更新前，向主人说明保护策略是否生效。

如果本地已有私有文件改动，AI 不得直接把它们全部 `git add .`。

### 阶段 1：处理本地提交

如果存在允许进入备份仓库的本地改动：

1. 只暂存“允许提交”的文件；
2. 私有保护文件继续保留为未提交状态；
3. 按脚本语义创建本地备份提交，默认提交说明可使用：
   - `本地修改备份`

如果没有允许提交的内容，就跳过这一步。

### 阶段 2：同步远程仓库

严格执行脚本语义：

1. `git fetch upstream --prune`
2. `git fetch origin --prune`
3. 同步 `upstream/main` 到 `origin/main`

如果远程连接失败，必须明确报错并停止，不得伪造“同步成功”。

### 阶段 3：预览并合并上游更新

合并前，AI 必须先展示：

1. 上游领先提交数量；
2. 最近若干条提交摘要；
3. 变更文件统计；
4. 哪些是可能影响本地保护目录的改动。

然后执行：

- `git merge upstream/main --no-edit`

如果是 fast-forward，也要明确说明。

### 阶段 4：冲突处理

发生冲突时，AI 必须先按规则自动分类，而不是统一停下。

#### 4.1 允许自动处理的冲突

默认直接接受上游版本：

- 所有 `.example` 文件；
- 插件目录中的非 `.env` 文件；
- 普通代码文件；
- 普通文档；
- 上游新增文件。

常见处理方式：

- `git checkout --theirs -- <file>`
- `git add <file>`

#### 4.2 必须暂停并询问主人的冲突

以下情况必须停下并汇报：

- `Agent/` 中已有文件的修改或删除；
- `TVStxt/` 中非 `.example` 文件的修改或删除；
- `dailynote/` 中无法明确判断归属的私有记忆文件；
- 主人临时额外指定的保护文件。

汇报时必须包含：

1. 冲突文件路径；
2. 冲突类型（`UU` / `DU` / `UD` / `AA` 等）；
3. 本地侧与上游侧各自代表什么；
4. 关键冲突片段；
5. 可以直接回复的决策选项。

禁止只说“有冲突，请选择”，不展示内容。

#### 4.3 插件 `.env` 的特殊规则

插件目录下的真实配置文件 `Plugin/*/config.env`：

- 默认不进入最终推送；
- 如果上游新增了这类文件，默认不要把它们一起推上去；
- 如果上游删除了对应插件，默认跟随上游删除仓库内版本；
- 如果主人明确说要保留本地配置，则只保留在本地，不强行进入提交。

#### 4.4 冲突全部解决后

AI 必须再次检查：

- `git diff --name-status --diff-filter=U`

只有在确认没有未解决冲突后，才能继续提交 merge。

### 阶段 5：提交合并结果

冲突全部解决后：

1. 提交 merge；
2. 记录 merge commit；
3. 明确说明这次 merge 是否包含主人的手动决策。

### 阶段 6：依赖检查

依赖处理必须严格跟随 `自动更新.bat` 的检查目标，不要自行扩大范围。

当前脚本关注：

- `Plugin/SciCalculator/requirements.txt`
- `Plugin/VideoGenerator/requirements.txt`
- `package.json`

处理原则：

1. 如果这些文件在本次 merge 前后有变化，则执行对应安装；
2. 如果没有变化，则跳过；
3. 允许记录安装告警，但不能把普通告警伪装成“更新失败”；
4. 如果安装命令确实失败，要把失败命令和错误原文告诉主人。

### 阶段 7：推送到个人备份仓库

推送前必须再次确认：

1. 私有保护文件仍未进入提交；
2. 当前分支是 `custom`；
3. 推送目标是 `origin custom`；
4. 需要推送的正是更新后的 merge 结果。

执行：

- `git push origin custom`

如果普通推送失败，才根据情况考虑：

- `git push origin custom --force-with-lease`

不得无理由直接强推。

---

## 7. 最终输出要求

更新完成后，AI 不能只说“已经更新完成”。必须输出以下内容：

### 7.1 结果摘要

至少包含：

- 是否完成上游合并；
- merge commit 是什么；
- 是否已推送到 `origin/custom`；
- 哪些本地保护文件没有被推送。

### 7.2 冲突处理记录

如果本次有冲突，必须说明：

- 哪些冲突自动按上游处理；
- 哪些冲突由主人拍板；
- 每项最终选了哪一边。

### 7.3 人类可读更新日志

必须使用清晰的自然语言总结，至少包括：

- 本次更新规模（提交数 / 主要目录）；
- 核心架构变化；
- 重要插件新增/增强；
- 被移除、停用、重构中的功能；
- 可能需要主人额外关注的兼容性或依赖问题。

不要只贴原始 commit 列表，要把它整理成人能读懂的摘要。

---

## 8. 推荐输出模板

AI 完成更新后，建议按下面结构输出：

1. **结果**
   - 是否合并成功
   - 是否推送成功
   - 哪些本地文件未推送

2. **冲突决议**
   - 文件路径
   - 决策结果
   - 是否由主人拍板

3. **依赖处理**
   - 哪些依赖文件发生变化
   - 执行了什么安装
   - 是否有告警

4. **本次更新内容摘要**
   - 核心变更
   - 新增功能 / 新插件
   - 删除或停用项
   - 风险与注意事项

---

## 9. AI 禁止事项

以下行为禁止再次发生：

- 未经核对就直接删除 `dailynote/`、`Agent/`、`TVStxt/` 中的主人内容；
- 在未建立保护清单前直接 `git add .`；
- 把真实 `.env` 配置提交进备份仓库；
- 遇到 `choice`/交互脚本卡住后直接放弃任务；
- 不展示冲突内容就要求主人做选择；
- 更新完成后不给出可读更新日志；
- 在未确认规则时擅自修改保护范围。

---

## 10. 一句话执行摘要

以后 AI 收到“按更新流程和 bat 更新”这类指令时，应自动理解为：

> **先读本指南与 `自动更新.bat`，按 bat 的阶段顺序等价执行；保护本地私有配置；对 `Agent/` 与 `TVStxt/` 的指定删改冲突停下汇报；所有 `.example` 跟上游；插件除真实 `.env` 外跟上游；更新完成后推送到 `origin/custom`，并输出人类可读更新日志。**
